The Three Lanes Are Categories, Not The Whole List

Almost every owner message falls into one of these lanes. The lane controls how much machinery turns on.

1. Thinking lane

You are exploring, questioning, proposing, worrying, or pressure-testing. This is where ideas stay cheap.

  • "I have an idea."
  • "Is this dumb?"
  • "What could go wrong?"
  • "Should we use an existing tool?"
2. Task lane

You want something moved into official work. Scope, authority, and proof requirements become explicit.

  • "Make this a task."
  • "Approved."
  • "Implement it."
  • "Fix this."
3. Verification lane

You want evidence, a second set of eyes, or a check that the worker is not inventing a green light.

  • "Verify this."
  • "Run doctor."
  • "Is Claude lying?"
  • "Where do we stand?"

General Work Map

This is the default Codex-session path when your message becomes real work. Small, reversible tasks may skip spec/ADR, but they do not skip repo truth or honest reporting.

Start Owner message Idea, bug, approval, verification request, or operational check.
Ground Read repo truth git status, AGENTS, PROJECT, ADRs, specs, and live .cockpit state.
Classify Choose lane Thinking, task, or verification. If consequences are unclear, pause before acting.
Bound Scope the work Objective, allowed scope, forbidden scope, acceptance criteria, proof needed.
Execute Do the smallest safe thing Edit, inspect, test, or hand off to worker depending on the lane.
Done? Evidence decides Tests, doctor, verifier verdict, CI exact SHA, and owner acceptance where needed.

Detailed Behind-The-Scenes Path

Owner
Says the thing: idea, approval, bug, suspicion, verification ask, commit/push ask, rollback ask, or release-readiness ask.
Codex
Checks the live repo before acting: current branch, dirty tree, existing instructions, current PCC task state, relevant docs, and whether the request is discussion-only or action-authorized.
Decision policy
If the choice affects architecture, scope, workflow, verification, or owner expectations, check docs/adr/. If it is a new significant decision, draft an ADR instead of silently deciding.
Spec policy
For a non-trivial feature or behavior change, write a one-screen spec under docs/specs/. Acceptance criteria need tests, not just prose.
PCC state
Official task work should be represented in canonical state and handoff artifacts: .cockpit/state/task-state.json, .cockpit/handoff/worker-directive.md, result files, and verification files.
Claude worker
When Claude is the worker, PCC drives it through supported non-interactive claude -p mode, not by wrapping the interactive terminal. Claude reads the directive and returns worker evidence. Worker claims are not final truth.
Local checks
Run the lightest sufficient checks first: focused tests, lint, script-contract checks, and scripts/doctor.ps1. Doctor is a health report, not a completion certificate.
Verifier
For non-trivial work, an independent verifier checks the exact diff/claim. PCC's primary verifier path is Codex CLI codex exec in read-only mode via verification scripts. Verdicts must say PASS, FAIL, INSUFFICIENT, BLOCKED, or OUT_OF_SCOPE.
Git gate
At commit time, staged changes are classified by stakes. T0/T1 changes require a valid diff-bound verification receipt or explicit bypass. The commit-msg hook can write a durable Verified-Receipt: trailer.
Push / CI
After push or PR, CI is the clean-machine fact. PCC should check CI for the exact SHA, not a stale stored claim. Release readiness uses the fresh release gate, not memory.

Full Utterance Inventory

These are the common things you can say and what they should trigger.

You say Codex should do Do not do
Idea Ground it in the roadmap, ADRs, current phase, and existing backlog. Keep it lightweight unless you ask to promote it. Do not start coding just because the idea is interesting.
Concern / suspicion Inspect the live files, state, or behavior that would prove or disprove it. Report uncertainty plainly. Do not soothe, assume, or declare "all good" without evidence.
Approved Convert the idea into bounded work: scope, acceptance criteria, allowed/forbidden areas, evidence, and task state/handoff if using PCC's worker lane. Do not let approval become a blank check.
Implement / fix Check repo truth, make scoped edits, protect unrelated dirty files, run relevant checks, and report proof plus residual risk. Do not self-certify non-trivial done.
Monitor Claude Inspect task state, directive, worker result, verifier result, git status, and doctor output. Look for role blur, stale handoff, fake-green, or drift. Do not trust Claude's prose over canonical files.
Check bridge Confirm app bridge code and tool status. If needed, run a tiny controlled claude -p probe, but only when bridge validation itself is the task. Do not casually burn model runs or treat a chat transcript as proof.
Run doctor Run scripts/doctor.ps1, summarize issues/warnings, and separate advisory warnings from blockers. Do not call doctor a release gate.
Verify Switch to evidence mode: inspect exact changes, run checks, judge claims, and return a verdict with "not proven." Do not fix during verification unless you explicitly ask for a repair pass.
Commit Review staged scope, run required checks, satisfy governance gate, commit with durable verification where required. Do not use --no-verify or bury unrelated changes.
Push / PR Push the intended branch, open/update PR if requested, then check exact-SHA CI truth. Do not claim remote green from stale local memory.
Release readiness Run the release gate fresh and treat PASS/FAIL/UNKNOWN honestly. Do not reuse yesterday's green.
Rollback / restore Pause, identify the exact target, prefer protected backups and non-destructive recovery, and ask before destructive moves. Do not reset or delete casually.

Bridge And Proof Boundaries

  • Claude communication in the app is via supported claude -p non-interactive mode with session continuity.
  • The file bridge is the truth surface: directives, results, task state, verification result, and evidence files.
  • Claude worker output is a claim. Verification decides whether the claim is supported.
  • Codex verification should be read-only and evidence-driven.
  • Doctor can reveal health problems but does not prove completion.
  • CI proves the exact pushed commit on a clean machine; it does not prove the owner wanted the right thing.
  • The owner remains the authority on "was this the right outcome?" even when tests prove "it was built as specified."

The anti-shitshow rule

If an automated owner-simulator or E2E run finds bugs, its job is to document and keep going when possible. It should not start fixing in place. Fixing turns the simulator into a worker, blurs the role, and risks losing the original goal.
